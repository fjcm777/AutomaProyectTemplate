Orden de arranque

1

store_db arranca

PostgreSQL 16 inicia y espera conexiones. Docker ejecuta pg_isready cada 10s.

2

store_backend arranca (cuando DB = healthy)

FastAPI + Uvicorn con --reload. Cambios en Python se recargan automáticamente.

3

store_frontend arranca

Vite dev server con HMR. Cambios en React/TypeScript actualizan el navegador al instante.

3. Backend — Flujo de una Petición HTTP

Cómo viaja una petición desde el cliente hasta la base de datos y de regreso

sequenceDiagram actor C as Cliente (Frontend) participant A as api.py
Router FastAPI participant S as service.py
Lógica de negocio participant R as repository.py
Acceso a datos participant DB as PostgreSQL C->>A: POST /api/v1/products/ Note over A: Valida payload con
ProductCreate (Pydantic)
Inyecta Session via get_db() A->>S: service.create_product(payload) Note over S: Verifica que category_id existe
Aplica reglas de negocio S->>R: repository.create(data) R->>DB: INSERT INTO products (...) DB-->>R: Fila creada R-->>S: Product ORM object S-->>A: Product ORM object Note over A: Serializa con ProductResponse
(response_model) A-->>C: 201 Created — JSON

Flujo completo para crear un producto. El mismo patrón aplica para GET, PUT y DELETE.

Responsabilidades por capa

api.py

Solo recibe la petición HTTP, valida el payload con Pydantic, inyecta dependencias y devuelve la respuesta. Sin lógica de negocio.

service.py

Contiene las reglas de negocio: validaciones cruzadas, integridad de FK, cálculos. Nunca toca la BD directamente.

repository.py

Único responsable de queries SQLAlchemy. Llama db.commit(). Sin lógica de negocio.

models.py

Mapea tablas de PostgreSQL a clases Python. Hereda TimestampMixin, Base para timestamps automáticos.

schemas.py

Define la forma de los datos de entrada (Create, Update) y salida (Response). Valida tipos con Pydantic.

7. Base de Datos — Modelo Entidad-Relación

Estado actual del esquema y campos de auditoría

erDiagram CATEGORIES { int id PK string name "UNIQUE" datetime created_at datetime updated_at } PRODUCTS { int id PK string name string description numeric price "10,2" string image_url int category_id FK datetime created_at datetime updated_at } CATEGORIES ||--o{ PRODUCTS : "tiene"

El campo price usa Numeric(10,2) para precisión exacta. Nunca Float.

Reglas del modelo de datos

TimestampMixin obligatorio

Todo modelo hereda TimestampMixin, Base. Los campos created_at y updated_at se gestionan automáticamente por la BD.

Precios con Numeric

Cualquier campo monetario usa Numeric(10, 2) en el modelo y Decimal en el schema. Serializar a float en la respuesta JSON con @field_serializer.

Migraciones con Alembic

Nunca modificar la BD manualmente. Siempre: cambiar el modelo → alembic revision --autogenerate → alembic upgrade head.

8. Reglas de Desarrollo

Convenciones a seguir en todo el proyecto

Backend

Separación de capas

api.py nunca llama a repository directamente. Siempre pasa por service.

Solo el repository hace commit

db.commit() únicamente en repository.py. Service y api no tocan la transacción.

Errores desde shared

Usar not_found() y bad_request() de shared/exceptions.py. Sin HTTPException directa en el código.

response_model siempre

Cada endpoint declara response_model. Nunca devolver un objeto ORM sin pasar por un schema Pydantic.

Registro en db/base.py

Todo modelo nuevo debe importarse en db/base.py para que Alembic lo detecte al generar migraciones.

Prefijos de router

Módulo en singular (product). Router con prefijo en plural (/products). Tags en plural también.

Frontend

React Query para servidor

Todo dato de la API vive en React Query. Nunca useState para datos remotos.

apiFetch centralizado

Toda llamada HTTP usa apiFetch de shared/api/clients.ts. Sin fetch directo en hooks o componentes.

Alias @/ para imports

Usar @/ para rutas absolutas desde src/. Nunca rutas relativas que suban más de un nivel.

queryKey consistente

Lista: ["entidad"]. Detalle: ["entidad", id]. Siempre array. Esencial para que el caché se invalide correctamente.

Solo Tailwind para estilos

Sin CSS inline ni archivos .css por componente. Clases condicionales con template literals o .join(" ").

Barrel exports

Cada feature expone su API pública vía index.ts. Otras features importan desde @/features/products, nunca desde rutas internas.
